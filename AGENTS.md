# AGENTS.md

> Instructions for AI agents (Codex, Claude Code, Cursor, WorkBuddy, etc.) working in this repository.
> Human contributors: see [README.md](./README.md).

## What this repo is

This is **ai-weekly-template** — a public template repository. End users **Fork** (or "Use this template") to spin up their own weekly newsletter site on GitHub Pages.

**Therefore everything in this repo is a SCAFFOLD, not personal content.** Keep it generic, keep it parameterized.

## Hard rules (in order of priority)

1. **Never put real, dated newsletter issues in `posts/` of THIS repo.** Only `example-issue.html` lives here as a reference. Real issues belong in users' Forked repos, not the template.
2. **Never hand-edit `posts/manifest.json`.** It is auto-generated by `.github/workflows/deploy.yml` on every push.
3. **Keep all user-facing strings parameterized** via `{{NEWSLETTER_TITLE}}` style placeholders in `index.html`. Do NOT hardcode "AI 行业周报" or similar — that's specific to the demo instance, not the template.
4. **Never invent source URLs** in `example-issue.html` or `prompts/weekly-prompt.md`. Use `https://example.com/` for placeholders, never plausible-looking fake URLs.
5. **Do not add a build step, JS framework, or backend.** The value of this template is its near-zero complexity. New features must be expressible in plain HTML + CSS + a single GitHub Actions workflow.

## File responsibilities

| File | Owner | Edit rules |
|---|---|---|
| `index.html` | template | Keep `{{PLACEHOLDER}}` tokens. No hardcoded brand strings. |
| `posts/example-issue.html` | template | Reference only. Use `https://example.com/` for all links. |
| `posts/manifest.json` | auto-generated | Never hand-edit. Keep the `posts: []` empty state. |
| `assets/style.css` | template | Single accent var `--green`. Don't add multiple accent colors. |
| `.github/workflows/deploy.yml` | template | Generic — site title/subtitle are "Weekly Newsletter" / "Updated weekly". |
| `prompts/weekly-prompt.md` | template | The opinionated authoring prompt. Domain-specific bits (S/A/B source list) are the secret sauce — preserve their structure when editing. |
| `README.md` | template | Public-facing onboarding doc. Quick Start MUST stay ≤ 5 steps. |
| `LICENSE` | template | MIT for code. Don't change without discussion. |

## Adding features

Before adding any feature, ask:
- Will every Fork user benefit, or only some?
- Does it require a JS framework / build step / backend? → reject
- Does it require knowing the user's specific domain (AI, climate, finance, etc.)? → parameterize
- Will it break existing Forks on upstream pull? → avoid; or document as a manual migration

## Repo-level commit conventions

- `feat: <what>` — new template feature
- `fix: <what>` — bug in template scaffolding
- `docs: <what>` — README / AGENTS.md
- `style: <what>` — CSS changes
- `chore: <what>` — workflow, config

## Things future agents will be tempted to do — DON'T

- ❌ Write a "real" recent newsletter issue into this repo. It's a template, not a publication.
- ❌ Replace `{{NEWSLETTER_TITLE}}` with anything specific.
- ❌ Add Tailwind / a framework / npm dependencies.
- ❌ Add tracking / analytics / external scripts to `index.html`.
- ❌ Edit `manifest.json` manually.
- ❌ Use plausible but fake URLs in the example issue.

## When unsure

Open an Issue and tag the maintainer. Do not make structural changes without discussion.
